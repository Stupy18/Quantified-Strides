# Story 016 — Alerts Migration & Disclaimer Footer

| Field | Value |
|---|---|
| **Story number** | 016 |
| **Type** | Feature |
| **Module** | Dashboard |
| **Layer(s) affected** | UI only (Web — Dashboard.jsx) |

---

## User Story

> As an **Athlete**, I want all training alerts to come from a single, authoritative source and I want any cycle-phase or health-adjacent recommendation to include the system disclaimer, so that I never miss a flag and I understand the limits of algorithmic advice.

---

## Context — Current State

The Dashboard's `AlertsSection` reads from the top-level `data.alerts` field, which is assembled independently by `alerts_service.py`. `DailyRecommendation.alerts` is a separate list produced inside the recommendation engine — it may contain overtraining flags, injury blocks, cycle-phase notices, and ACL-awareness entries that are not currently surfaced anywhere on the Dashboard. The two alert sources are not merged. Additionally, `DailyRecommendation.disclaimer` (a plain-text string, non-null when cycle-phase modifiers are applied or when confidence is 'low'/'no_data') is not displayed.

---

## New Behavior

**Alert source consolidation:**

The Dashboard's alert display merges both sources:
- `data.alerts` (existing anomaly/overtraining alerts from `alerts_service.py`)
- `DailyRecommendation.alerts` (recommendation-engine flags: safety gate fires, cycle-phase notices, ACL-awareness entries, confidence warnings)

Deduplication: if the same alert text appears in both lists, it is shown once. Ordering: `data.alerts` entries first, then `DailyRecommendation.alerts` entries.

Alert types from `recommendation.alerts` use the same visual treatment as existing alerts (pill badge or banner, styled by severity if a severity field is present; otherwise default styling).

**Disclaimer footer:**

When `DailyRecommendation.disclaimer` is non-null, a disclaimer line renders as a small, muted-text footer below the full recommendation section (below the Exercise Block or Running Prescription, whichever is last). The disclaimer is not inside a card — it sits as plain text, visually de-emphasised, so it is present but does not compete with the prescription content.

When `disclaimer` is null, the footer is not rendered.

**`rec.avoid` field retired:**

The old `rec.avoid` field (freeform string of things to avoid) is no longer read. Its content is now surfaced through `recommendation.alerts`. No migration of historical data is needed — `rec.avoid` was UI-only display state.

---

## Out of Scope

- Changing alert severity logic or adding new alert types (intelligence layer concern — Stories 003, 010)
- Persisting dismissed alerts across sessions
- Mobile implementation

---

## Acceptance Criteria

- [ ] Alerts section renders entries from both `data.alerts` and `DailyRecommendation.alerts`, merged and deduplicated
- [ ] `data.alerts` entries appear before `recommendation.alerts` entries
- [ ] No duplicate alert text is shown when both sources contain the same entry
- [ ] When `disclaimer` is non-null, it renders as muted-text footer below the recommendation section
- [ ] When `disclaimer` is null, no footer element is rendered
- [ ] `rec.avoid` is no longer referenced anywhere in the component
- [ ] Regression: existing `data.alerts` display is visually unchanged when `recommendation.alerts` is an empty list
- [ ] Regression: Dashboard loads correctly when `disclaimer` is null and `recommendation.alerts` is an empty list

## Technical Notes

Relevant files: `frontend/src/pages/Dashboard.jsx` → `AlertsSection` component. After Story 004, `recommendation.alerts` is an array of strings (or structured alert objects — confirm shape in Story 004's output contract). `disclaimer` is a string field on `DailyRecommendation`. Merge logic: `[...data.alerts, ...recommendation.alerts].filter((v, i, a) => a.indexOf(v) === i)` if both are plain string arrays. Adjust if alert objects have an `id` or `text` field.
