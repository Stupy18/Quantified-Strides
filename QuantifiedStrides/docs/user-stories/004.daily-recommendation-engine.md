# Story 004 — Daily Recommendation Engine & Typed Output Contract

| Field | Value |
|---|---|

| **Type** | Feature |

---

## User Story

> As an **Athlete**, I want a daily training recommendation that reflects all my current signals and has a typed, reliable structure, so that I can trust the suggestion and developers can build UI on a stable contract.

---

## Context — Current State

`intelligence/recommend.py` returns a freeform dict with no stable schema. The function accepts keyword arguments for individual signals — callers pass whatever they happen to have. There is no readiness composite score. Session type selection is minimal and not tied to `session_type_catalog`. The dashboard service reads the dict without validation. `recommendation_service.py` does not map to a typed model.

---

## New Behavior

**Input contract (BREAKING CHANGE):**
- `build_recommendation()` accepts a single `signals: dict` parameter
- All 26 keys from `RECOMMENDATION_PROTOCOL.md §3.1` must be present; missing values as `None`, no key may be omitted
- Omitting a key raises `KeyError` with the missing key name
- Keyword argument signature removed

**Readiness composite score:**
- Weighted average: HRV z-score (0.4), sleep readiness (0.3), TSB normalized to [−30, +20] (0.2), `daily_readiness` subjective (0.1)
- Missing signals: weight denominator reduced proportionally, remaining weights renormalized to sum to 1.0

**Sport and session type selection (two independent matrices — see `RECOMMENDATION_PROTOCOL.md §5.2`):**

Aerobic and strength session intensity are gated by **different signals**. TSB measures aerobic fatigue only and must never gate strength prescription. Mixing them was the old design; this is the corrected version.

- Active `training_plans` scheduled session type takes precedence over priority-weight scoring
- When no active plan: sport selected via `user_profile.primary_sports` priority weights modulated by muscle freshness, weather, time_available, and hard rules

**Aerobic matrix — gated by `readiness_level` + TSB:**
- `readiness_level == 'peak'` AND `tsb > 5` → intervals / threshold eligible
- `readiness_level == 'peak'` AND `tsb ≤ 5` → tempo / aerobic (quality, not intensity)
- `readiness_level == 'good'` AND `tsb ≥ −5` → tempo / aerobic
- `readiness_level == 'good'` AND `tsb < −5` → Z2 aerobic base
- `readiness_level == 'moderate'` → Z2 only
- `readiness_level in ('low', 'rest')` → Z1 / rest

An athlete in a normal 8-week base-building block will have `tsb < 5` for most of it — this is expected. The aerobic matrix handles this by capping at tempo/aerobic rather than intervals.

**Strength matrix — gated by `readiness_level` + `pattern_fatigue_residuals`. TSB is NOT used here:**
- `readiness_level in ('peak', 'good')` AND `pattern_residual < 1.5` (fresh) AND `hrv_z ≥ −0.5` → heavy compound, full goal intensity
- `readiness_level in ('peak', 'good')` AND `pattern_residual 1.5–3.0` (fatigued) → moderate strength
- `readiness_level in ('peak', 'good')` AND `pattern_residual > 3.0` (depleted) → light / deload
- `readiness_level == 'moderate'` → light strength regardless of residuals
- `readiness_level in ('low', 'rest')` → skip or mobility only

The HRV floor (`hrv_z ≥ −0.5`) on the heavy compound row catches systemic CNS fatigue that pattern residuals (movement-specific) cannot see. If HRV is suppressed, cap at moderate even when residuals are fresh.

**Pattern fatigue residuals — three-tier cold-start fallback:**
1. `pattern_fatigue_ledger` has ≥ 5 entries for this pattern → use ledger-based exponential decay (normal path)
2. `pattern_fatigue_ledger` has < 5 entries for this pattern BUT `strength_sets` data exists (user backlogged sessions) → use muscle_freshness approximation: `residual_approx = (1 − mean(freshness[primary_muscles])) × 3.0` (per spec §7.0.2)
3. BOTH ledger and `strength_sets` are empty for this pattern (true cold start — user has never logged a session for this movement) → default to `moderate`, never `heavy`

Do **not** collapse these three cases into one. A user who backlogs 6 months of strength history on sign-up is not a cold-start user; they have data. Only case 3 triggers the conservative default.

**Priority when both aerobic and strength signal "peak ready":**
- Aerobic session is always evaluated first
- Strength timing is governed by `compute_strength_timing()` (§8b): aerobic session today → strength `after_run_3h_min`; no aerobic → strength `anytime`
- Both sessions are included in the recommendation on the same day; the conflict resolution is sequencing (§5.3), not exclusion

**Typed output — `DailyRecommendation` Pydantic model:**
- Fields: `sport`, `session_type`, `intensity_zone` (int 1–5), `duration_min` (int), `hr_ceiling_bpm` (int | None), `notes` (str), `disclaimer` (str), `alerts` (list[Alert]), `exercise_suggestions` (list[ExerciseSuggestion] | None), `running_prescription` (RunningPrescription | None), `readiness_composite` (float | None), `confidence` ('high'|'medium'|'low'|'no_data')
- `hr_ceiling_bpm`: pre-computed HR ceiling for today's session — `max_hr × zone_boundary_pct`, minus early-luteal offset when applicable. `None` when `max_hr` is unknown.
- Return type is always `DailyRecommendation` — never `None`, never a plain dict

**Confidence and degradation:**
- ≥ 4 signals available → `'high'`; 2–3 → `'medium'`; 1 → `'low'`; 0 → `'no_data'`
- New user with zero history → Zone 2 base session, `confidence='no_data'`, disclaimer set

**Service layer:**
- `recommendation_service.py` maps `DailyRecommendation` fields to the dashboard Pydantic model
- Raw `DailyRecommendation` object is never passed to the API layer directly
- `dashboard_service.py` updated to consume the typed mapping

## Acceptance Criteria

- [ ] `build_recommendation(signals)` with all 26 keys (some `None`) executes without error
- [ ] `build_recommendation(signals)` with a missing key raises `KeyError` naming the missing key
- [ ] Return value is always an instance of `DailyRecommendation`; never `None` or `dict`
- [ ] Gate 0 fires → return is `DailyRecommendation` with `session_type='rest'`, not an exception
- [ ] `readiness_composite` with all 4 signals present is a weighted average in [0, 1]
- [ ] `readiness_composite` with HRV `'no_data'` renormalizes remaining weights to sum to 1.0; does not return `None` unless all signals are missing
- [ ] Active plan's scheduled session type overrides priority-weight scoring
- [ ] `confidence='no_data'` when no signals are available; Zone 2 base session recommended
- [ ] `GET /dashboard` response includes all `DailyRecommendation` fields in the dashboard schema; no raw Pydantic model leakage
- [ ] `dashboard_service.py` and `recommendation_service.py` compile without type errors after the dict→model migration
- [ ] Generate Tests
