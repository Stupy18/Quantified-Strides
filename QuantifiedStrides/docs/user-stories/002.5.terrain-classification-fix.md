## User Story

> As an **Athlete**, I want terrain to be classified correctly based on how I labeled my activity on my Garmin device, so that my road and trail biomechanics baselines are not contaminated by misclassified sessions.

---

## Context — Current State

`classify_terrain()` in `intelligence/analytics/biomechanics.py` uses only `workout_metrics.gradient_pct` to determine whether a session was road or trail:

```python
def classify_terrain(gradient_series: list[float]) -> str:
    if len(gradient_series) < 10:
        return 'unknown'
    pct_graded  = sum(1 for g in gradient_series if abs(g) > 3) / len(gradient_series)
    grad_range  = max(gradient_series) - min(gradient_series)
    return 'trail' if (pct_graded > 0.25 or grad_range > 15) else 'road'
```

This is causally wrong. Gradient measures **topography** (elevation change). The biomechanics difference between road and trail (GCT +12–18%, cadence −8–12 spm at identical HR effort) is caused by **surface irregularity and footing unpredictability** — not gradient. These are orthogonal dimensions.

**Failure cases of the current implementation:**
- A road hill-repeat session (high gradient, paved surface) → classified as `trail`, contaminating trail baseline with road-like GCT/cadence values.
- A flat forest singletrack (low gradient, irregular footing) → classified as `road`, contaminating road baseline with trail-like values.

`workouts.sport` already carries the correct primary signal: Garmin records `'trail_run'` and `'running'` as distinct activity types, set by the athlete on the device before recording begins.

---

## New Behavior

`classify_terrain()` uses `workouts.sport` as the primary classifier and gradient as fallback only:

```python
def classify_terrain(sport: str, gradient_series: list[float]) -> str:
    if sport == 'trail_run':
        return 'trail'
    if sport == 'running':
        if len(gradient_series) >= 10:
            pct_graded = sum(1 for g in gradient_series if abs(g) > 3) / len(gradient_series)
            grad_range = max(gradient_series) - min(gradient_series)
            if pct_graded > 0.25 or grad_range > 15:
                logger.warning(
                    'terrain_override: sport=running classified as trail by gradient '
                    '(pct_graded=%.2f, grad_range=%.1f) — verify activity labeling',
                    pct_graded, grad_range,
                )
                return 'trail'
        return 'road'
    return 'unknown'
```

The gradient fallback handles the edge case where an athlete runs a trail but forgets to switch the activity type on their watch. The warning log creates an audit trail for labeling drift.

**On first deploy: backfill + recompute required** (see Acceptance Criteria).

## Acceptance Criteria

- [ ] `classify_terrain()` signature updated to accept `sport: str` as first argument
- [ ] All call sites updated to pass `workouts.sport`
- [ ] `sport == 'trail_run'` always returns `'trail'`, regardless of gradient values
- [ ] `sport == 'running'` returns `'road'` by default; returns `'trail'` only when gradient criteria are met, with a warning logged
- [ ] `sport` values other than `'running'` and `'trail_run'` return `'unknown'`
- [ ] Backfill migration: run `UPDATE workouts SET terrain_type = classify_terrain(sport, gradient_series)` for all existing rows (via one-off script or Flyway migration)
- [ ] After backfill: recompute `biomechanics_baselines` for all users
- [ ] Post-backfill query confirms distribution: `SELECT sport, terrain_type, COUNT(*) FROM workouts WHERE sport IN ('running','trail_run') GROUP BY sport, terrain_type` — near-zero `trail_run` rows with `terrain_type = 'road'` and vice versa
- [ ] Post-recompute: per-terrain mean GCT and cadence in `biomechanics_baselines` match notebook-confirmed direction (trail GCT higher, trail cadence lower vs road)
- [ ] Warning log line emitted when gradient overrides sport label
- [ ] Generate Tests
