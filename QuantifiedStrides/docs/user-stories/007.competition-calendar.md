---

## User Story

> As an **Athlete**, I want to log my upcoming races with priority and have the system automatically account for them when making training recommendations, so that my training builds toward my goal events.

---

## Context — Current State

The `competitions` table does not exist (added in Story 001). There is no API for creating or retrieving competitions. The recommendation engine has no concept of days-to-competition or taper trigger — it recommends training with no awareness of race proximity.

---

## New Behavior

**Competition CRUD:**
- `POST /api/v1/competitions` — create a competition with `event_date`, `sport`, `event_name`, `priority` (A/B/C), `distance_km`, `notes`
- `GET /api/v1/competitions` — list all future competitions for the authenticated user, ordered by `event_date` asc
- Priority A = goal race; B = tune-up; C = training race

**`days_to_competition` signal (in-memory, per request):**
- Nearest upcoming A-priority competition → `days_to_competition` count + `competition_priority = 'A'`
- No A-priority → nearest B-priority used
- No future competitions → both are `None`; plan generator operates in open-ended mode

**Taper trigger:**
- A-priority competition within `taper_weeks × 7` days (per active plan's phase structure) → session duration reduced; quality session frequency drops to 1/week
- B-priority competition: no taper applied; training continues on normal phase schedule

---

## Out of Scope

- UI for the competition calendar (Profile page integration is a separate frontend story)
- Automatic race discovery from Garmin
- C-priority races modifying any recommendation behavior in v1

## Acceptance Criteria

- [ ] `POST /api/v1/competitions` creates a `competitions` row; returns 201 with the created object
- [ ] `GET /api/v1/competitions` returns only future competitions for the authenticated user, ordered by `event_date` asc
- [ ] `days_to_competition` is the correct calendar day count to the nearest A-priority future competition
- [ ] When no A-priority competition exists, `days_to_competition` uses nearest B-priority; when neither exists, both are `None`
- [ ] Taper trigger fires only for A-priority competitions within the taper window; B-priority does not trigger taper
- [ ] Recommendation engine correctly reads `days_to_competition = None` as no-plan-anchor signal; no error raised
- [ ] Generate Tests
