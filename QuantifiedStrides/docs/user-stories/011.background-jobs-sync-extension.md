# Story 011 — Background Jobs & Garmin Sync Extension

| Field | Value |
|---|---|
| **Story number** | 011 |
| **Type** | Feature |
| **Module** | Garmin Sync / Recommendation Engine |
| **Layer(s) affected** | API/service / Intelligence layer |

---

## User Story

> As an **Athlete**, I want training load metrics, zone speeds, HRV baseline, and 1RM estimates to update automatically without manual intervention, so that the recommendation engine always works from fresh, accurate data.

---

## Context — Current State

`ingestion/workout.py` ingests workout data but computes nothing post-ingest (no TRIMP, no terrain classification, no biomechanics columns). ATL/CTL/TSB are computed on-the-fly per request. There are no scheduled background jobs anywhere in the codebase. `user_profile.zone_speeds`, `user_profile.hrv_baseline_mean/sd`, and `user_1rm_cache` do not yet exist (added in Story 001).

---

## New Behavior

**Post-sync pipeline (triggered by `POST /api/v1/sync/garmin`):**
- For each newly ingested workout: compute `trimp`, `terrain_type`, `fatigue_index`, `hr_stability_last_10min` and write to `workouts` — before `training_load_daily` is updated
- If sync fails mid-pipeline: no partial writes to `training_load_daily`; prior row preserved

**Monthly background job — zone speed calibration:**
- `compute_zone_speeds()` run for each user with ≥ 5 new HR-stable qualifying runs since last calibration
- Upserts `user_profile.zone_speeds`; skips user if threshold not met (existing calibration retained)

**Continuous HRV baseline re-establishment:**
- Triggered after each new `sleep_sessions` insert where preceding day TRIMP ≤ 50 or `NULL`
- `establish_hrv_baseline()` runs; updates `user_profile.hrv_baseline_mean/sd` only when ≥ 14 clean readings available
- First-time baseline write occurs at exactly the 14th qualifying reading

**1RM cache invalidation (event-driven):**
- Triggered on every new `strength_sets` insert with reps ≤ 10
- `user_1rm_cache` for `(user_id, exercise_id)` deleted and reinserted via Epley formula
- reps > 10 → cache unchanged

**Weekly taper TSB projection (background job, weekly):**
- For each user with an active training plan in Peak phase: compute projected TSB at race day using `project_race_day_tsb(current_tsb, split_key, remaining_peak_weeks)`
- Result stored or cached for use in split conflict alert payloads
- Job skipped when no user has an active Peak-phase plan

**Daily open-ended deload evaluation (background job, daily):**
- For each user in open-ended mode: increment `training_plans.open_ended_load_week_counter` at week boundary
- Evaluate 6 reactive deload triggers per `RECOMMENDATION_PROTOCOL.md §15.1` (ACWR threshold, TSB floor, ramp rate, check-in feel, HRV, sleep slope)
- When any trigger fires or counter reaches 5: insert a deload week, reset counter to 0
- Job writes to `training_plan_weeks`; does not modify completed past weeks

**Split re-evaluation on strength_goal change (event-driven):**
- Triggered when `training_plans.strength_goal` is updated via the profile API
- Re-evaluates all future `training_plan_weeks` rows where `split_key_overridden = FALSE` using the new goal
- Clears conflict-driven overrides (`split_key_overridden = TRUE AND split_override_acknowledged = TRUE`) that are no longer conflicting under the new goal
- Clears `split_phase_conflict` alerts that no longer apply
- Does not touch weeks where athlete manually overrode split for non-conflict reasons

**Backfill scripts (one-time, run on deploy):**
- `scripts/backfill_trimp.py` — compute TRIMP for all existing workouts with HR data
- `scripts/backfill_training_load.py` — populate `training_load_daily` for all users from existing TRIMP series
- `scripts/backfill_terrain_type.py` — classify terrain for all existing workouts with `workout_metrics`

---

## Out of Scope

- Quarterly max HR update job (deferred — low priority for current data volume)
- Real-time streaming of load metrics to the UI
- Multi-server/Celery job distribution (APScheduler in-process is sufficient for single-server deployment)

---

## Acceptance Criteria

- [ ] After a successful Garmin sync, `workouts.trimp`, `workouts.terrain_type`, `workouts.fatigue_index`, and `workouts.hr_stability_last_10min` are all populated for new workouts before the endpoint returns
- [ ] `workouts.trimp = NULL` for workouts without `avg_hr`; no error raised
- [ ] `training_load_daily` is upserted for the syncing user after sync success; unchanged if sync fails mid-pipeline
- [ ] Monthly zone speed job does not update `user_profile.zone_speeds` when user has < 5 new qualifying runs
- [ ] `user_profile.hrv_baseline_mean/sd` is `NULL` until exactly 14 clean readings exist, then written on the 14th
- [ ] Subsequent clean readings after baseline is established → baseline recomputed and updated
- [ ] `strength_sets` insert with reps ≤ 10 → `user_1rm_cache` invalidated and reinserted
- [ ] `strength_sets` insert with reps > 10 → `user_1rm_cache` unchanged
- [ ] APScheduler monthly job registered in FastAPI lifespan; verifiable via server startup log
- [ ] `backfill_trimp.py` runs to completion without error on existing data; `workouts.trimp` populated for all rows with `avg_hr`
- [ ] Dashboard load time after backfill reads from `training_load_daily` (O(1)); does not recompute from TRIMP series
- [ ] Weekly taper TSB projection job runs for all Peak-phase users; skips users not in Peak phase
- [ ] Open-ended deload counter increments at week boundary; deload week inserted when counter reaches 5 or a reactive trigger fires; counter resets to 0 after any deload
- [ ] All 6 reactive deload triggers evaluated daily per spec §15.1; at least one trigger firing correctly inserts a deload week
- [ ] `strength_goal` change triggers split re-evaluation; non-overridden future weeks adopt the new goal's eligible split; stale conflict overrides are cleared
- [ ] `strength_goal` change does not touch weeks where split was manually overridden for non-conflict reasons
- [ ] Generate Tests

## Technical Notes

Relevant spec: `RECOMMENDATION_PROTOCOL.md §12`. APScheduler registered in FastAPI lifespan (`main.py`). Post-sync compute calls added to `ingestion/workout.py`. Backfill scripts added to `scripts/`. 1RM cache invalidation can be wired as a SQLAlchemy event listener on `strength_sets` insert or called explicitly from `services/strength_service.py`. HRV baseline trigger wired in `services/sleep_service.py` after sleep session write.
