## User Story

> As a **developer**, I want all recommendation engine v2.0 schema additions in place so that every P1 feature build starts on the correct data model without incremental schema debt blocking downstream work.

---

## Context ‚Äî Current State

`RECOMMENDATION_PROTOCOL.md ¬ß2` defines 18 new tables, 4 ALTER TABLE extensions to existing tables, and 4 new columns on `workouts` (from ¬ß3's signal registry). None of these exist today. The current schema (V005) has no `training_load_daily`, no plan tables, no movement pattern ontology, no competition calendar, and no signal columns on `workouts` or `user_profile`. Every P1 feature in ¬ß13 depends on at least one of these additions being present before implementation can start.

---

## New Behavior

There is no user-visible behavior change. After this story:

**New tables created (18):**
- `training_load_daily` ‚Äî precomputed ATL/CTL/TSB/ACWR/ramp_rate per user per day
- `competitions` ‚Äî competition calendar (A/B/C priority races)
- `menstrual_cycles` ‚Äî cycle log for female athlete protocol
- `biomechanics_baselines` ‚Äî terrain-stratified per-athlete biomechanics baselines
- `user_1rm_cache` ‚Äî Epley 1RM estimates, invalidated per new set
- `movement_patterns` ‚Äî movement pattern catalog, seeded with 9 patterns
- `exercise_relationships` ‚Äî ontology graph edges (SUBSTITUTES, PROGRESSES_TO, etc.)
- `pattern_fatigue_ledger` ‚Äî per-athlete, per-pattern fatigue load post-session
- `exercise_session_log` ‚Äî CSP solver selection log per session
- `plan_types` ‚Äî plan type catalog (marathon, 5k, strength, etc.)
- `plan_goal_tiers` ‚Äî target time ‚Üí tier + volume parameters
- `plan_type_phases` ‚Äî Base/Build/Peak/Taper fractions per type + tier
- `session_type_catalog` ‚Äî session type definitions (zone, duration, recovery)
- `plan_session_templates` ‚Äî day-of-week session assignments per plan type/tier/phase, with parity trigger
- `strength_phase_catalog` ‚Äî strength phase definitions, seeded with 4 phases (gpp, spp, power, maintenance)
- `strength_phase_exercises` ‚Äî exercise assignments per strength phase
- `training_plans` ‚Äî one active plan per user (partial unique index)
- `training_plan_weeks` ‚Äî instantiated week rows per plan

**Columns added to existing tables:**
- `user_profile`: `training_status`, `sex`, `date_of_birth`, `max_hr`, `max_hr_source`, `zone_speeds`, `hormonal_contraception`, `hrv_baseline_mean`, `hrv_baseline_sd`
- `exercises`: `primary_pattern`, `secondary_patterns`, `equipment`, `cns_cost`, `local_fatigue_cost`, `skill_level`, `velocity_based`, `bilateral`, `tags`, `enabled`, `permanent_fixture`
- `injuries`: `severity`, `affected_activities`, `cross_training_ok`, `cleared_by_user`, `return_volume_pct`
- `workouts`: `trimp`, `terrain_type`, `fatigue_index`, `hr_stability_last_10min`
- `training_plans`: `strength_goal`, `goal_weights`

**Triggers:**
- `trg_plan_session_templates_parity` on `plan_session_templates` ‚Äî enforces that a slot uses either `'any'` or the `odd/even` pair, never both

## Out of Scope

- Populating any of the new tables with data (no backfill, no seeding beyond the two catalogs below)
- Seeding `plan_types`, `plan_goal_tiers`, `plan_type_phases`, `session_type_catalog`, `plan_session_templates`, `strength_phase_exercises` ‚Äî these are P2 (plan) catalog seeds per ¬ß13
- Any application code change ‚Äî no Python, no API, no frontend
- Computing or writing values to `training_load_daily` ‚Äî that is Story 002
- `exercise_relationships` population ‚Äî no seed data defined in ¬ß2; table created empty

- [ ] All migrations apply cleanly on top of an existing V005 DB ‚Äî no destructive changes to existing rows or columns

- [ ] `strength_phase_catalog` contains exactly 4 seed rows: `gpp`, `spp`, `power`, `maintenance` per ¬ß2
- [ ] `trg_plan_session_templates_parity` trigger is active and raises an exception when a parity conflict is inserted
- [ ] `training_plans_one_active_per_user` partial unique index exists (enforces one active plan per user)
- [ ] All `ALTER TABLE` additions use `IF NOT EXISTS` ‚Äî idempotent on repeated runs
- [ ] New columns on existing tables have defaults that do not break any existing query (no NOT NULL without DEFAULT on populated tables, except where ¬ß2 explicitly defines a safe default)
- [ ] `workouts.trimp`, `workouts.terrain_type`, `workouts.fatigue_index`, `workouts.hr_stability_last_10min` added ‚Äî nullable, no existing rows affected
- [ ] No committed migration file is edited ‚Äî new versions only, per `WORKFLOW.md`
- [ ] Flyway checksums pass on re-run (no file modifications after first apply)

## Technical Notes

Relevant spec: `RECOMMENDATION_PROTOCOL.md ¬ß2` ‚Äî all DDL is written out verbatim and can be copied directly into migration files.

Current highest Flyway version: **V005**. New migrations start at `V006`. The implementer may split ¬ß2 across multiple files (e.g., one per logical group) or combine into fewer files ‚Äî either is valid as long as dependency order is respected (e.g., `movement_patterns` must exist before `exercises` FK to `primary_pattern`; `plan_types` before `plan_goal_tiers`; `competitions` before `training_plans`).

Relevant paths: `QuantifiedStrides/db/flyway/` ‚Äî all `.sql` files in this directory are picked up by Flyway in version order.

`training_plans.strength_goal` and `goal_weights` are defined in ¬ß2 via ALTER TABLE after the CREATE TABLE ‚Äî these can be included directly in the CREATE TABLE in the migration to avoid two operations on the same new table.
