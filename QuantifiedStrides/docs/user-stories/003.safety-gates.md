# Story 003 — Safety Gates

| Field | Value |
|---|---|
| **Story number** | 003 |
| **Type** | Feature |
| **Module** | Recommendation Engine |
| **Layer(s) affected** | Intelligence layer |

---

## User Story

> As an **Athlete**, I want the system to enforce hard safety overrides before any session is recommended, so that I am never prescribed training that is clinically unsafe given my current load, injury state, or overtraining risk.

---

## Context — Current State

`intelligence/recommend.py` performs no safety gate logic. There are no ACWR thresholds enforced programmatically — a user with ACWR > 1.8 would currently receive a normal recommendation. Injuries in the `injuries` table are stored but not read by the recommendation engine. The `STANDARD_DISCLAIMER` does not exist anywhere in the codebase.

---

## New Behavior

**Gate evaluation order — first gate that fires short-circuits all others:**

**Gate 0 — Critical (ACWR > 1.8):**
- Output: `session_type='rest'`, alert level `'ots'`
- Alert message appends: *"Consider consulting a healthcare provider."*
- `STANDARD_DISCLAIMER` always present in output

**Gate 1 — Danger (ACWR > 1.5, ≤ 1.8):**
- Output constrained to Zone 1 active recovery, duration ≤ 45 min
- No quality sessions, no strength sessions permitted

**Gate 2 — Caution (ACWR > 1.3, ≤ 1.5 AND ramp_rate > 10%/week):**
- Recommended duration capped at 60 min
- Intensity not further restricted

**Injury block (evaluated after ACWR gates):**
- All activities in `injuries.affected_activities` blocked when `cleared_by_user = FALSE`
- `cross_training_ok = TRUE` → cycling/swimming cross-training offered as alternative
- `cross_training_ok = FALSE` → rest only; no endurance or strength permitted
- `affected_activities = NULL` → system infers from `injury_type`; if ambiguous, all high-impact blocked, user prompted via alert

**Standard disclaimer:**
- `DailyRecommendation.disclaimer = STANDARD_DISCLAIMER` on every output, regardless of gate state

---

## Out of Scope

- UI display of alerts and disclaimer — separate frontend story
- Modifying the injury table or creating injuries via this story — that is Story 009
- ACWR thresholds becoming user-configurable (fixed per spec)

---

## Acceptance Criteria

- [ ] ACWR > 1.8 → `session_type='rest'`, OTS alert, "Consider consulting a healthcare provider." appended, disclaimer set
- [ ] ACWR > 1.5 and ≤ 1.8 → Zone 1 recovery only, ≤ 45 min; no strength or quality session eligible
- [ ] ACWR > 1.3 and ramp_rate > 10 → duration capped at 60 min; zone not restricted
- [ ] Gate 0 fires → injury block is not separately evaluated (short-circuit confirmed)
- [ ] Active injury with `cleared_by_user = FALSE` → blocked activities never appear in recommendation
- [ ] `cross_training_ok = TRUE` → at least one alternative activity is offered in the recommendation
- [ ] `cross_training_ok = FALSE` → `session_type='rest'`, no alternative offered
- [ ] `STANDARD_DISCLAIMER` present in every `DailyRecommendation` output, including normal days with no gates fired
- [ ] When ACWR is `NULL` (insufficient data), ACWR gates are skipped entirely — no false gate fires
- [ ] `apply_safety_gates()` is called before any sport/session-type selection logic runs
- [ ] Generate Tests

## Technical Notes

Relevant spec: `RECOMMENDATION_PROTOCOL.md §4`. Create `intelligence/safety_gates.py` with `apply_safety_gates(signals) -> GateResult`. `GateResult` indicates which gate fired (or `None`) and the override recommendation fields. Relevant existing files: `intelligence/recommend.py` (integration point), `repos/recommendation_repo.py` (reads active injuries). `training_load_daily` (from Story 002) is the source for ACWR and ramp_rate.
