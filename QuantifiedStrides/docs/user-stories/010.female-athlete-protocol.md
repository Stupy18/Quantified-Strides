# Story 010 — Female Athlete Protocol

| Field | Value |

---

## User Story

> As an **Athlete**, I want the system to optionally account for my menstrual cycle phase when making training recommendations, so that my prescriptions reflect my body's natural performance windows without medical overreach.

---

## Context — Current State

`user_profile` has no `sex` or `hormonal_contraception` column (added in Story 001). `menstrual_cycles` table does not exist (added in Story 001). No cycle phase estimation logic exists anywhere. All language constraints (performance-optimisation framing only) are new.

---

## New Behavior

**Cycle phase estimation (in-memory, per request):**
- Only when `user_profile.hormonal_contraception IS NOT TRUE`
- Phase estimated as `(today - start_date) % predicted_cycle_length` from nearest `menstrual_cycles` row within 60 days
- Returns `None` when no record within 60 days or `hormonal_contraception = TRUE` → no modifiers applied

**Intensity modifiers per phase:**
- `'follicular'`: no modifier; quality sessions fully eligible
- `'ovulatory'`: no modifier (baseline)
- `'luteal'`: zone cap reduced by 1; duration reduced 10%; note added advising RPE-based pacing
- `'menstrual'`: Zone 1–2 default; quality sessions still eligible if `readiness_composite > 0.75`
- All modifiers are performance-optimisation framing — language never diagnostic

**ACL awareness flag (non-blocking):**
- `user_profile.sex = 'female'` AND CSP solver selects a plyometric or COD exercise → `DailyRecommendation.alerts` includes ACL-awareness entry
- Flag does not modify exercise selection or block any exercise
- Not added when session has no plyometric or COD exercises

**Garmin menstrual data import:**
- Garmin sync returns menstrual cycle JSON → upserted to `menstrual_cycles` with `source='garmin'`
- Manual entry endpoint → `source='manual'`

---

## Out of Scope

- Diagnosing or predicting cycle irregularities
- Any change to recommendations when `hormonal_contraception = TRUE` (system is fully neutral)
- UI for cycle tracking (Profile page integration is a separate frontend story)

---

## Acceptance Criteria

- [ ] `cycle_phase = None` when `hormonal_contraception = TRUE` — no modifiers applied, no error
- [ ] `cycle_phase = None` when no `menstrual_cycles` row within 60 days — no modifiers applied
- [ ] `'luteal'` phase → intensity zone cap reduced by 1 and duration reduced 10% in recommendation output
- [ ] `'menstrual'` phase + `readiness_composite > 0.75` → quality session remains eligible (not blocked)
- [ ] `'menstrual'` phase + `readiness_composite ≤ 0.75` → Zone 1–2 only
- [ ] `user_profile.sex = 'female'` + plyometric exercise selected → ACL-awareness alert present in `DailyRecommendation.alerts`
- [ ] ACL-awareness alert text is non-diagnostic, awareness-only language
- [ ] `user_profile.sex ≠ 'female'` → no ACL-awareness flag, regardless of exercises selected
- [ ] Garmin sync with menstrual cycle JSON → `menstrual_cycles` row upserted with `source='garmin'`
- [ ] `STANDARD_DISCLAIMER` always present in cycle-adjusted recommendations
