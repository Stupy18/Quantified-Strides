# Story 009 — Injury Management

| Field | Value |

---

## User Story

> As an **Athlete**, I want to log an injury and have the system automatically adjust my recommendations — blocking unsafe activities, offering cross-training, and guiding me back to full training — so that I recover without losing fitness or making things worse.

---

## Context — Current State

The `injuries` table exists but has no `severity`, `affected_activities`, `cross_training_ok`, `cleared_by_user`, or `return_volume_pct` columns (all added in Story 001). The recommendation engine does not read injuries. `training_plans.training_state` does not exist (added in Story 001). There is no return-to-run protocol anywhere in the codebase.

---

## New Behavior

**Injury state machine (via `training_plans.training_state`):**
- `'active'` → `'injured'` when a new `injuries` row is inserted with `severity` and `affected_activities`
- `'injured'` → `'cross_training'` automatically when `cross_training_ok = TRUE`
- `'cross_training'` or `'injured'` → `'return_to_run'` when `injuries.cleared_by_user = TRUE`
- `'return_to_run'` → `'active'` once return ramp is complete

**Activity blocking (enforced by safety gates — Story 003):**
- All activities in `injuries.affected_activities` blocked while `cleared_by_user = FALSE`
- `affected_activities = NULL` → system infers from `injury_type`; if ambiguous, all high-impact activities blocked, user alerted

**Cross-training:**
- `cross_training_ok = TRUE` → cycling or swimming at equivalent Zone 2 duration; CNS budget for strength reduced
- `cross_training_ok = FALSE` → rest only

**Return-to-run volume ramp:**
- Triggered when `cleared_by_user = TRUE`; `return_volume_pct` computed at clearance from pre-injury CTL-implied weekly km
- Week 1: `return_volume_pct × pre_injury_weekly_km`; Zone 1–2 only
- Each subsequent symptom-free week (no new `injuries` row in past 7 days): volume increases 10%; Zone 1–2 maintained

**API:**
- `POST /api/v1/injuries` — log a new injury
- `PATCH /api/v1/injuries/{id}/clear` — set `cleared_by_user = TRUE`, trigger return-to-run ramp computation

---

## Out of Scope

- UI for injury logging
- Medical advice or injury diagnosis (all language is load-management framing only)
- Automatic injury detection from biomechanics signals (future story)

---

## Acceptance Criteria

- [ ] `POST /api/v1/injuries` creates an injury row and transitions `training_plans.training_state` to `'injured'`
- [ ] `PATCH /api/v1/injuries/{id}/clear` sets `cleared_by_user = TRUE`, transitions state to `'return_to_run'`, and computes `return_volume_pct`
- [ ] Blocked activity never appears in recommendation while `cleared_by_user = FALSE`
- [ ] Alert in `DailyRecommendation.alerts` explains which activity is blocked and why
- [ ] `cross_training_ok = TRUE` → at least one alternative activity recommended; CNS budget reduced
- [ ] `cross_training_ok = FALSE` → `session_type='rest'`; no endurance or strength offered
- [ ] Week 1 return volume equals `return_volume_pct × pre_injury_weekly_km`; session type is Zone 1–2
- [ ] Symptom-free week 2: recommended volume is week 1 × 1.10; Zone 1–2 maintained
- [ ] New `injuries` row in past 7 days during `'return_to_run'` → progression paused, volume held at current level
- [ ] `affected_activities = NULL` → high-impact activities blocked and alert prompts user to clarify
- [ ] Generate Tests
