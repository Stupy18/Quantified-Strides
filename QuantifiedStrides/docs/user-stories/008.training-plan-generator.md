# Story 008 — Training Plan Generator

| Field | Value |

---

## User Story

> As an **Athlete**, I want to generate a structured training plan anchored to a goal race (or open-ended if I have no race), so that the daily recommendation follows a periodized schedule instead of reacting only to today's signals.

---

## Context — Current State

No training plan infrastructure exists. `training_plans`, `training_plan_weeks`, and all catalog tables (`plan_types`, `plan_goal_tiers`, `plan_type_phases`, `session_type_catalog`, `plan_session_templates`, `strength_phase_catalog`) are added in Story 001 but not yet used. The recommendation engine operates purely reactively — there is no concept of Base/Build/Peak/Taper phases.

---

## New Behavior

**Plan creation:**
- `POST /api/v1/plans` — creates a plan from `plan_type_key`, optional `competition_id`, `target_time_min`, `strength_goal`
- Linked competition → `plan_mode = 'race_anchored'`, `race_date` set from competition; `training_plan_weeks` generated from `plan_start_date` to `race_date`
- No competition → `plan_mode = 'open_ended'`; rolling Base→Build with no Peak/Taper; extends until abandoned or competition linked
- One active plan per user enforced (partial unique index)

**Goal tier determination:**
- Target time bracketed against `plan_goal_tiers.max_time_min`
- `target_time_min = None` → catchall tier (NULL `max_time_min` row) used

**Weekly template generation:**
- `training_plan_weeks` rows created from `plan_session_templates` for the `(plan_type_key, tier, phase, quality_count, week_parity, day_of_week)` combination
- Strength sessions assigned per `plan_type_phases.strength_phase_key`, scheduled on non-quality-run days
- Week 4 of each mesocycle marked `cutback_week = TRUE`, volume reduced 30% from peak

**Phase progression:**
- Phase (Base/Build/Peak/Taper) determined from `plan_type_phases.phase_fraction` relative to `plan_start_date` and `race_date`
- `GET /api/v1/plans/active` returns current plan including this week's phase and session type

**Strength goal integration:**
- `strength_goal = 'sport_support'` → CSP solver CNS budget at minimum, `CARRY_BRACE`/`ISOLATION` prioritized
- `strength_goal = 'combination'` + `goal_weights` → weights must sum to 1.0 ± 0.01; validated at creation
- `strength_goal` defaults to `'sport_support'` when `None`

---

## Out of Scope

- Plan editing (modifying weeks after generation) in v1
- UI for plan creation and visualization
- Automatic plan revision on injury — that is Story 009

---

## Acceptance Criteria

- [ ] `POST /api/v1/plans` with a valid `competition_id` creates a race-anchored plan with `training_plan_weeks` rows from today to `race_date`
- [ ] `POST /api/v1/plans` without `competition_id` creates an open-ended plan with no `race_date`
- [ ] Creating a second active plan while one is already active returns 409 (conflict)
- [ ] Correct tier is selected from `plan_goal_tiers` given `target_time_min`; catchall tier used when `target_time_min = None`
- [ ] Week 4 of a 4-week mesocycle has `cutback_week = TRUE` and `target_volume_km` reduced 30% from week 3
- [ ] Phase of a mid-plan week is correct per `phase_fraction` calculation
- [ ] `goal_weights` with values not summing to 1.0 ± 0.01 is rejected at plan creation with a validation error
- [ ] Active plan's scheduled session type takes precedence over priority-weight sport selection in the recommendation engine
- [ ] `GET /api/v1/plans/active` returns 404 when no active plan exists
