## User Story

> As an **Athlete**, I want to generate a structured training plan anchored to a goal race (or open-ended if I have no race), so that the daily recommendation follows a periodized schedule instead of reacting only to today's signals.

---

## Context ‚Äî Current State

No training plan infrastructure exists. `training_plans`, `training_plan_weeks`, and all catalog tables (`plan_types`, `plan_goal_tiers`, `plan_type_phases`, `session_type_catalog`, `plan_session_templates`, `strength_phase_catalog`) are added in Story 001 but not yet used. The recommendation engine operates purely reactively ‚Äî there is no concept of Base/Build/Peak/Taper phases, split selection, or periodized deload scheduling.

---

## New Behavior

### Plan Creation

- `POST /api/v1/plans` ‚Äî creates a plan from `plan_type_key`, optional `competition_id`, `target_time_min`, `strength_goal`
- Linked competition ‚Üí `plan_mode = 'race_anchored'`, `race_date` set from competition; `training_plan_weeks` generated from `plan_start_date` to `race_date`
- No competition ‚Üí `plan_mode = 'open_ended'`; rolling Base‚ÜíBuild with no Peak/Taper; extends until abandoned or competition linked
- One active plan per user enforced (partial unique index)

### Goal Tier Determination

- Target time bracketed against `plan_goal_tiers.max_time_min`
- `target_time_min = None` ‚Üí catchall tier (NULL `max_time_min` row) used

### Weekly Template Generation

- `training_plan_weeks` rows created from `plan_session_templates` for the `(plan_type_key, tier, phase, quality_count, week_parity, day_of_week)` combination
- Strength sessions assigned per `plan_type_phases.strength_phase_key`, scheduled on non-quality-run days using the week's active `split_key`
- Week 4 of each mesocycle marked `cutback_week = TRUE`, volume reduced 30% from peak

### Phase Progression

- Phase (Base/Build/Peak/Taper) determined from `plan_type_phases.phase_fraction` relative to `plan_start_date` and `race_date`
- `GET /api/v1/plans/active` returns current plan including this week's phase and session type

### Strength Goal Integration

- `strength_goal = 'sport_support'` ‚Üí CSP solver CNS budget at minimum, `CARRY_BRACE`/`ISOLATION` prioritized
- `strength_goal = 'combination'` + `goal_weights` ‚Üí weights must sum to 1.0 ¬± 0.01; validated at creation
- `strength_goal` defaults to `'sport_support'` when `None`

### Split Catalog and Session Pattern Mapping

The split catalog defines the supported strength session structures. It is seeded data ‚Äî never edited by users or updated at runtime. Schema revisions ship as new Flyway migrations.

**`split_catalog` (seeded, 6 rows):**

| split_key | min_days | max_days | sessions | concurrent_risk | volume_freq | max_concurrent_aerobic_days |
|---|---|---|---|---|---|---|
| `full_body_maintenance` | 1 | 1 | 1 | 1 | 1 | 7 |
| `upper_only_maintenance` | 1 | 1 | 1 | 1 | 1 | 7 |
| `full_body` | 1 | 3 | 1‚Äì3 | 1 | 2 | 5 |
| `upper_lower` | 2 | 4 | 2 | 2 | 3 | 5 |
| `push_pull_legs` | 3 | 6 | 3 | 3 | 4 | 3 |
| `push_pull_lower_upper` | 4 | 5 | 4 | 2 | 5 | 4 |

`concurrent_risk`: 1 = low (easy to schedule around running), 2 = moderate, 3 = high (dedicated leg day directly competes with quality run recovery).

| split_key | Base | Build | Peak (`sport_support`) | Peak (hybrid) | Taper |
|---|---|---|---|---|---|
| `full_body_maintenance` | ‚úó | ‚úó | ‚úó | ‚úó | ‚úì forced |
| `upper_only_maintenance` | fallback only | fallback only | fallback only | fallback only | fallback only |

### Split Selection ‚Äî Algorithm Suggests, User Can Override

Split selection is **algorithmic by default, user-overridable**. The system computes the best split from athlete data and phase context; the athlete can declare a preference that the algorithm respects when valid.

**`user_profile.preferred_split TEXT NULL`** ‚Äî null = fully algorithmic. When set, the algorithm returns the preferred split if it is eligible for the current phase and gym day count. The preference is validated at each weekly computation ‚Äî it is not blindly honoured.

**`split_key` lives on `training_plan_weeks`**, not `training_plans`. Split is a week-level attribute computed at plan generation time and refreshed weekly. This allows the split to adapt when phase transitions, gym availability changes, or `strength_goal` is updated.

**`select_split()` ‚Äî called at week generation:**

```python
def select_split(
    rolling_gym_days_4w: float,   # 4-week rolling avg from strength_sessions; falls back to
                                   # user_profile.gym_days_week when < 4 weeks of history
    phase: str,                   # 'onboarding'|'base'|'build'|'peak'|'taper'
    strength_goal: str,
    aerobic_days_per_week: int,   # running sessions this week from training_plan_weeks
    preferred_split: str | None,  # from user_profile
) -> str:

    if phase == 'taper':
        return 'full_body_maintenance'

    eligible = [s for s in SPLIT_CATALOG
                if s.min_days <= rolling_gym_days_4w <= s.max_days
                and s.max_concurrent_aerobic_days >= aerobic_days_per_week
                and phase in s.eligible_phases_for_goal(strength_goal)]

    if not eligible:
        return 'upper_only_maintenance' if aerobic_days_per_week >= 5 else 'full_body'

    if preferred_split and any(s.split_key == preferred_split for s in eligible):
        return preferred_split

    def score(s: SplitEntry) -> int:
        if strength_goal == 'sport_support':
            return -s.concurrent_risk_score
        return s.volume_frequency_score

    return max(eligible, key=score).split_key
```

`rolling_gym_days_4w` cold-start: when `COUNT(strength_sessions) < 4` (beginner ‚Äî see Onboarding section below), inherit `user_profile.gym_days_week` directly. The rolling average is not meaningful without at least 4 weeks of logged history.

### Phase-Split Conflict Alert and Override Mechanism

When `select_split()` would select a different split than the athlete's `preferred_split` due to phase ineligibility, a **`split_phase_conflict` alert** is generated.

**Alert behaviour by athlete type:**

*`strength_goal = 'sport_support'` in Peak phase (PPL or PPLU preferred):*
- Alert fires: "Your preferred split includes a dedicated leg day. Switching to Upper/Lower for Peak phase to protect leg freshness for race day."
- Athlete sees two options: accept switch (default) or keep preferred split
- If athlete keeps preferred split: **consequence disclaimer must be acknowledged** before confirming

*`strength_goal in ('combination', 'hypertrophy', 'strength')` in Peak phase:*
- No blocking alert. Split is eligible by default.
- Informational TSB projection surfaced passively: "Your leg day X days before the race is estimated to cost ~Y TSB points (projected race-day TSB: Z). This is the concurrent training tradeoff you have chosen."

**Consequence disclaimer** uses `project_race_day_tsb()` to compute the TSB delta from the leg session's estimated TRIMP:

> "Keeping Push/Pull/Legs through Peak phase adds a high-volume leg session within [X] days of your race. Projected race-day TSB: [Y] ‚Üí [Z] (optimal window: 5‚Äì15). Residual neuromuscular fatigue from leg sessions within 7 days of an A-race does not fully resolve before the start line. This is your choice."

**Schema fields on `training_plan_weeks`:**

```sql
split_key                    TEXT    NOT NULL,
split_key_overridden         BOOLEAN DEFAULT FALSE,   -- user manually set this week
split_override_acknowledged  BOOLEAN DEFAULT FALSE,   -- disclaimer accepted
split_override_tsb_cost      FLOAT   NULL,            -- projected TSB delta shown in disclaimer
gym_days_override            SMALLINT NULL            -- week-specific gym day count; NULL = inherit profile
```

### Beginner Onboarding Phase

**Detection:** An athlete is a beginner if `COUNT(strength_sessions) < 4` at plan creation. This is the same threshold as the pattern fatigue cold-start (Story 004 ¬ß7.0.2) ‚Äî below 4 sessions, no movement baseline exists.

**Onboarding block:** Two `training_plan_weeks` rows with `phase = 'onboarding'` are prepended to the plan before Base begins. Normal phase eligibility, split selection, and session intensity logic do not apply during these weeks.

**What an onboarding session is:**
- `split_key = 'full_body'` always (all movement patterns covered)
- Weight target: RPE 5‚Äì6, never to failure
- One working set per movement pattern family (not per exercise)
- CSP solver selects exercises with the lowest `skill_level` within each pattern (beginner-appropriate)
- The athlete logs RPE per set as normal

**Diagnostic output:** After two weeks the system derives a pattern weakness profile from logged RPE at given weights. Patterns where the athlete recorded high RPE (‚â• 7) at relatively low absolute load flag as primary development targets. This profile is stored in `user_profile.pattern_weakness_profile JSONB` and used by the CSP solver to weight exercise selection in the first Base mesocycle.

**Transition:** When both onboarding weeks are completed, `user_profile.strength_onboarding_complete = TRUE` is set and the plan generator creates the first Base mesocycle week using the normal `select_split()` call with `rolling_gym_days_4w` now seeded from onboarding session count.

### Mid-Plan `strength_goal` Change

When `user_profile.strength_goal` is updated while a plan is active, the following happens for all **future unexecuted** `training_plan_weeks`:

1. **Re-evaluate non-overridden splits:** Re-run `select_split()` for all weeks where `split_key_overridden = FALSE`. Write updated `split_key` values.

2. **Clear conflict-driven overrides where the conflict no longer exists:** For weeks where `split_key_overridden = TRUE` and `split_override_acknowledged = TRUE` ‚Äî if the new `strength_goal` makes the previously-conflicting split eligible (e.g., athlete moves from `sport_support` to `combination`, making PPL in Peak valid), clear `split_key_overridden = FALSE`, `split_override_acknowledged = FALSE`, `split_override_tsb_cost = NULL`. The override is no longer meaningful; the algorithm reclaims the decision.

3. **Clear stale conflict alerts:** Any unacknowledged `split_phase_conflict` alerts derived from the old `strength_goal` are removed from the active alerts queue.

4. **Surface notification:** "Your strength goal has been updated to [new goal]. Your split schedule has been refreshed for upcoming weeks."

Past weeks (already executed) are never modified.

### Open-Ended Plan Deload Triggers

Open-ended plans (`plan_mode = 'open_ended'`) have no Peak or Taper phase. Deload weeks are inserted reactively by the system rather than pre-scheduled by phase fraction.

**Trigger 1 ‚Äî Fixed cadence (5 weeks):** Every 5 weeks, a deload week is unconditionally inserted. Equivalent to a 4:1 loading-to-deload ratio. `training_plans.open_ended_load_week_counter INT DEFAULT 0` is incremented weekly and reset to 0 after each deload. A reactive deload (any trigger below) also resets the counter ‚Äî no double deload.

**Trigger 2 ‚Äî ACWR overreach:** ACWR ‚â• 1.5 for 3 consecutive days ‚Üí deload week inserted within the next 7 days.

**Trigger 3 ‚Äî Deep TSB:** TSB ‚â§ ‚àí20 for 3 consecutive days ‚Üí deload week inserted within the next 7 days.

**Trigger 4 ‚Äî Ramp rate sustained:** CTL increase > 3 pts/week for 2 consecutive weeks ‚Üí deload.

**Trigger 5 ‚Äî Morning check-in degradation:** `daily_readiness.overall_feel ‚â§ 2` for 3 consecutive days ‚Üí deload. This is the primary subjective override ‚Äî consecutive poor check-ins indicate accumulated fatigue the objective signals may not yet reflect.

**Trigger 6 ‚Äî HRV suppression sustained:** HRV z-score < ‚àí1.5 for 3 or more consecutive mornings ‚Üí deload. The threshold is lower than Gate 2 (which fires at 5 days for race-anchored plans) because open-ended plans have no race to protect ‚Äî earlier intervention is appropriate.

**Trigger 7 ‚Äî Sleep deficit cluster:** 3-day rolling sleep quality below the athlete's personal baseline threshold (existing alert) ‚Üí deload.

**What a deload week means for open-ended plans:**
- Volume reduced 30% from current week
- Aerobic intensity capped at Zone 2
- Strength session: `split_key = 'full_body_maintenance'`, no quality sessions
- Duration: 1 week; counter resets after

### TSB-Anchored Taper Volume

Fixed-percentage tapers (75% ‚Üí 60% ‚Üí 45%) are athlete-blind. `compute_taper_volume_schedule(pre_taper_ctl, pre_taper_atl, taper_weeks, peak_km)` uses binary search to find the first-week volume fraction that projects race-day TSB onto `target_tsb=12.0` (midpoint of the optimal [5, 20] window). Each subsequent taper week is 15% lower. No week drops below 40% of peak (neuromuscular maintenance floor).

`assign_weekly_volumes()` accepts optional `pre_taper_ctl` and `pre_taper_atl`. When provided, taper weeks use `compute_taper_volume_schedule()`. When `None` (plan preview before first sync ‚Äî CTL/ATL not yet available), falls back to fixed percentages.

`project_race_day_tsb(current_ctl, current_atl, daily_trimp, days)` simulates ATL/CTL forward via EWMA. Used at plan creation and weekly during taper to detect drift.

TSB target window:
- `< 5`: under-tapered ‚Äî residual fatigue blunts performance
- `5‚Äì15`: optimal ‚Äî target midpoint is 12
- `15‚Äì20`: acceptable for recreational athletes
- `> 20`: over-tapered ‚Äî CTL decay outweighs freshness gain

### Gate 4 ‚Äî TSB-Aware Pre-Competition Week Volume

- `tsb ‚â• 12` ‚Üí `volume_multiplier = 0.80`
- `tsb ‚â• 7` ‚Üí `volume_multiplier = 0.70`
- `tsb < 7` ‚Üí `volume_multiplier = 0.60`

`current_tsb` is already a gate input ‚Äî no additional query.

### Weekly Taper Projection Background Job

Runs every Monday for users with `plan_mode='race_anchored'` and `training_state='tapering'`. Reads current CTL/ATL, calls `project_race_day_tsb()`:

- Projected TSB < 5 with ‚â• 14 days to race ‚Üí alert: "Taper under-tracking ‚Äî reduce volume or add easy days"
- Projected TSB > 20 with ‚â• 14 days to race ‚Üí alert: "Taper over-tracking ‚Äî add a short quality stimulus this week"
- ‚â§ 13 days to race ‚Üí informational TSB projection only; no plan revision

Projected TSB written to `training_plans.projected_race_day_tsb` (updated weekly).
