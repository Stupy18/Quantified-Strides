---

## User Story

> As an **Athlete**, I want the system to compute all training signals from my Garmin data, HRV, and strength sessions, so that the recommendation engine always has an accurate, current picture of my readiness and load.

---

## Context — Current State

Signal assembly is scattered across `intelligence/training_load.py` and `intelligence/recovery.py` and is partially incomplete. TRIMP is not yet computed per workout — `workouts.trimp` does not exist. There is no `training_load_daily` table, so ATL/CTL/TSB are recomputed from raw workouts on every dashboard request. HRV baseline is computed from a rolling window (contaminated by training stress). Pattern fatigue residuals and biomechanics fatigue index are not implemented. No centralised signal assembly function exists.

---

## New Behavior

**TRIMP (post-sync, per workout):**
- Computed using Banister's sex-specific formula: `duration_min × HRr × k × exp(y × HRr)` where `k=0.86, y=1.67` for female, `k=0.64, y=1.92` for male/prefer_not_to_say
- HRr clamped to [0, 1]
- Written to `workouts.trimp`; `NULL` when `avg_hr` or `max_hr` is missing — session excluded from load metrics

**ATL / CTL / TSB / ACWR / ramp_rate (post-sync, precomputed):**
- Written to `training_load_daily` (one row per user per day) after each successful Garmin sync
- TAU_ATL = 7, TAU_CTL = 42 (HEURISTIC — see `RECOMMENDATION_PROTOCOL.md §3.3`)
- ACWR is `NULL` when CTL < 1; ramp_rate is `NULL` when < 14 days of data exist
- < 3 sessions in TRIMP series → ATL/CTL/TSB written as 0.0, ACWR `NULL`

**HRV baseline and z-score (continuous accumulation):**
- Baseline computed only from clean readings: preceding day TRIMP `NULL` (rest) or ≤ 50 (easy Z2)
- Stored in `user_profile.hrv_baseline_mean` and `hrv_baseline_sd`; `NULL` until ≥ 14 clean readings
- `compute_hrv_status()` returns `'suppressed'` (z < −1.5), `'normal'`, `'elevated'` (z > +1.0), or `'no_data'`

**Sleep readiness (in-memory, per request):**
- Derived from most recent `sleep_sessions` row; `None` when no record within past 2 days

**Biomechanics fatigue index (post-sync):**
- Weighted deviation of cadence (50%), GCT (30%), vertical ratio (20%) from `biomechanics_baselines`
- Written to `workouts.fatigue_index`; `NULL` when no baseline exists for that session's terrain type

**Zone speeds calibration (monthly background job):**
- Qualifying runs = HR-stable final 10 min (CV < 0.05) — stored as `workouts.hr_stability_last_10min`
- Pace ranges per HR zone stored in `user_profile.zone_speeds` JSONB; zone omitted when < 5 qualifying runs

**Pattern fatigue residuals (in-memory, per request):**
- Exponential decay per pattern from `pattern_fatigue_ledger` using `movement_patterns.fatigue_decay_tau_h`
- Cold-start (< 5 ledger entries for a pattern): approximated as `(1 − mean(freshness[primary_muscles])) × 3.0`

**All signals centralised in `intelligence/signal_assembly.py → assemble_signals()`:**
- Returns complete signals dict with all keys; missing signals as `None` — no key omitted

## Acceptance Criteria

- [ ] `workouts.trimp` is computed and written for every newly ingested workout with `avg_hr`; `NULL` otherwise
- [ ] `workouts.hr_stability_last_10min` is computed post-sync for running workouts; `NULL` when < 10 HR readings in final 10 min
- [ ] `training_load_daily` is upserted for the syncing user after every successful `POST /api/v1/sync/garmin`; prior row is not modified if sync fails mid-pipeline
- [ ] ACWR is `NULL` when CTL < 1; ramp_rate is `NULL` when < 14 days of data
- [ ] `compute_hrv_status()` returns `'no_data'` when `user_profile.hrv_baseline_mean` is `NULL`; HRV safety gates are skipped in this state
- [ ] HRV baseline only includes readings where preceding TRIMP ≤ 50 or `NULL`; it does NOT include post-hard-session readings
- [ ] `sleep_readiness` is `None` (not 0) when no `sleep_sessions` row within 2 days
- [ ] `workouts.fatigue_index` is `NULL` when `biomechanics_baselines` has no matching terrain row; no error is raised
- [ ] `assemble_signals()` returns a dict with every key from the §3.1 signal registry; missing values are `None`, no key is omitted
- [ ] `user_profile.zone_speeds` is not updated when user has < 5 new qualifying runs since last calibration
- [ ] Pattern fatigue cold-start approximation (< 5 ledger entries) is used only until threshold is crossed; ledger-based decay takes over above 5 entries
- [ ] Dashboard load time does not increase — ATL/CTL/TSB is read from `training_load_daily`, not recomputed on request
